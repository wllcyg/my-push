# 什么是 Elasticsearch 的倒排索引？为什么它这么快？

在了解倒排索引（Inverted Index）之前，我们先来看看什么是正向索引，这样更容易理解它的优势。

## 1. 正排索引（Forward Index）
正排索引是我们最常见的索引方式，比如关系型数据库（如 MySQL）中未加全文索引的普通文本字段。它的结构是 **“文档 ID -> 文档内容”**。
当我们想要查找包含某个关键词的文档时，由于我们只有文档 ID 到内容的映射，数据库必须遍历每一个文档，检查内容中是否包含该关键词。这就好比在一本没有目录的厚书中找特定的一个词，只能一页一页地翻，效率极低（也就是全文扫描）。

## 2. 什么是倒排索引（Inverted Index）
倒排索引是搜索引擎（如 Elasticsearch、Lucene）的核心数据结构。它的结构颠倒了过来，变成了 **“关键词 -> 文档 ID 列表”**。

它的建立过程类似于：
1. 取出所有文档的内容。
2. 将内容进行**分词**，提取出所有的“词（Term）”。
3. 记录每个词出现在哪些文档中，形成映射关系。

**举个直观的例子：**
假设我们有三个文档：
- 文档 1 (ID: 1)：`Elasticsearch is fast`
- 文档 2 (ID: 2)：`Elasticsearch is a search engine`
- 文档 3 (ID: 3)：`Lucene is fast`

经过分词和倒排索引构建后，大致结构如下：

| 词（Term） | 出现的文档 ID 列表 (Posting List) |
| :--- | :--- |
| elasticsearch | [1, 2] |
| is | [1, 2, 3] |
| fast | [1, 3] |
| a | [2] |
| search | [2] |
| engine | [2] |
| lucene | [3] |

当用户搜索 "fast" 时，ES 会直接去倒排索引中查找 "fast" 这个词，瞬间就能得出它在文档 1 和文档 3 中，直接返回结果，而不需要去扫描所有文档的内容。这就是它被称为“倒排”（从词找文档）的原因。

---

## 3. 倒排索引为什么快？
如果只是上面的表格，在海量数据（几十亿个词）的情况下，查找依然会遇到瓶颈。倒排索引之所以在海量数据下依然能保持极高的检索性能，归功于底层极致的结构优化：

### 3.1 词典（Term Dictionary）和 词典索引（Term Index）
如果我们有上千万个不同的词，怎么快速在字典里找到我们要的词呢？
- **Term Dictionary（词典）**：ES 会将所有的 Term 进行字典序排序。由于排过序，就可以使用二分查找法，时间复杂度为 O(log N)。
- **Term Index（词典索引）**：即便用了二分查找，如果词典存在磁盘上，大量随机读取磁盘依然很慢。ES 引入了 Term Index。它是一个基于 **FST (Finite State Transducer)** 的前缀树结构，就像是一本字典的“按字母首字母分类的目录”。这本目录被极致压缩，体积非常小，**可以直接全量加载到内存中**。
  
> **查找过程**：先在内存中通过 Term Index 快速定位到该词在磁盘 Term Dictionary 中的大概位置（Block 段），然后再去磁盘进行极少量的读取，极大减少了磁盘 I/O。

### 3.2 倒排表（Posting List）的压缩
Posting List 记录了包含某个词的全部文档 ID。如果一个词（比如 "的", "is"）非常常见，它的 Posting List 会非常长，占用大量内存和磁盘。ES 对此进行了极致的压缩：
- **增量编码（Delta Encoding）**：假设文档 ID 列表是 `[73, 300, 302, 332, 343, 372]`。转化为增量后变成 `[73, 227, 2, 30, 11, 29]`。这样所有的数字都变得很小，所需的存储位数（bit）就变少了。
- **FOR / Roaring Bitmaps 压缩**：经过增量编码后的数据，ES 还会采用 Frame Of Reference (FOR) 或 Roaring Bitmaps（咆哮位图）算法进行进一步的位级别压缩，从而可以用极小的空间存储海量的文档 ID。

### 3.3 快速的交集与并集计算（跳表 Skip List）
在实际搜索中，用户通常会输入多个词，比如搜索 `"Elasticsearch fast"`。
这时候 ES 需要分别找到 "Elasticsearch" 的 Posting List `[1, 2]` 和 "fast" 的 Posting List `[1, 3]`，然后求它们的**交集**，从而找出同时包含这两个词的文档。
为了加速这个合并过程，ES 为长长的 Posting List 建立了**跳表（Skip List）**。通过跳表，指针可以大跨度地跳过大量不相关的 ID，将复杂度从 O(N) 大幅降低，实现非常高效的集合求交/求并操作。

## 4. 搜出来的结果怎么排名？（BM25 算法）
倒排索引解决了“**怎么快速找到包含关键词的文档**”的问题，但如果你搜到了 100 篇文章，ES 怎么决定哪篇排在第一，哪篇排在最后呢？
这就是 **相关性算分算法** 的工作了。Elasticsearch 目前默认使用的算分算法叫做 **BM25 (Best Matching 25)**。

当你搜索一个词（比如“混合检索”被拆分为“混合”、“检索”）时，BM25 主要通过以下三个核心维度来给每一篇文章打分（也就是你在查询结果里看到的 `_score`）：

### 4.1 词频（TF - Term Frequency）
**核心思想：词在文章中出现的次数越多，越相关。**
如果文章 A 里“检索”出现了 10 次，文章 B 里只出现了 1 次，那显然文章 A 更相关。
- **BM25 的优化（词频饱和度）**：早期的 TF-IDF 算法中，词频越高分数无限增加。但 BM25 认为，当一个词出现次数达到一定数量后（比如 20 次），再多出现几次（比如 25 次）对相关性的提升应该越来越小，最终趋于平缓。

### 4.2 逆文档频率（IDF - Inverse Document Frequency）
**核心思想：越稀有的词，价值越大。**
假如你搜“Elasticsearch 的原理”。“的”这个字在几乎所有的文章里都有，而“Elasticsearch”这个词只在少数几篇文章里有。
- 如果一篇文章命中了“的”，不能说明它跟你的搜索有多大关系。
- 如果一篇文章命中了“Elasticsearch”，那它非常可能就是你想找的。
BM25 会降低常见词的权重，**大幅提升稀有词的权重**。这就是为什么只命中罕见词的文章，得分往往比命中了几个常见词的文章还要高。

### 4.3 字段长度归一化（Field Length Normalization）
**核心思想：越短的文本匹配上，相关性越强。**
假如你搜“RAG”：
- 文章 A：标题（一共 10 个字）里包含“RAG”。
- 文章 B：长篇大论（一共 10000 个字）的段落里包含了一个“RAG”。
很显然，文章 A 专门在标题里提到了它，一定更相关！BM25 会惩罚包含该词的长文本，**奖励短文本**。

> **总结一句话：** 如果一个**很罕见的词**，在一篇**很短的文本**里，出现了**很多次**，那么这篇文档的 BM25 得分就会突破天际，稳居第一！

---

## 总结：
Elasticsearch 的倒排索引之所以快，本质上是因为：
1. **结构优势**：把“遍历文档找词”变成了“用词直接命中相关文档”。
2. **内存级索引**：利用 FST 结构的 Term Index 在内存中极速定位磁盘数据，有效避免了大量磁盘 I/O。
3. **极致压缩**：对海量的文档 ID 列表进行增量编码和位图压缩，极大地节省了空间和传输成本。
4. **高效合并**：利用 Skip List 加速多条件查询时的结果集合并（交集、并集）速度。
