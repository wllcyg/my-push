# 售后智能 RAG 知识库系统 - PRD

> 文档状态：草稿 v0.1
> 项目性质：个人学习项目（技术实践 + 简历作品）
> 最后更新：2026-07-01

---

## 一、项目背景

### 1.1 为什么做这个项目

汽车售后场景的知识库有几个典型难点，是通用客服 RAG 场景不常遇到的：

- 知识来源分散在多个异构系统（维修手册、技术公告、备件库存、工单记录、经验帖）
- 不同角色（客服、技师、工程师）需要看到的知识粒度完全不同
- 部分知识涉及密级（未公开质量问题、安全相关维修方案），不能简单地"检索到就能看"
- 错误的维修建议有安全和法律风险，答案必须可追溯、可验证

本项目通过搭建一个简化但结构完整的售后知识库系统，实践 **RAG 检索 + 权限治理** 的组合设计，作为个人技术学习与简历作品。

### 1.2 项目目标

- 跑通一个可用的售后知识问答 MVP：能检索、能问答、能溯源
- 实现基于角色和密级的权限过滤，且过滤发生在检索阶段（而非生成后打码）
- 验证多知识源混合检索（向量检索 + 结构化数据查询）的可行架构
- 沉淀一套可复用的权限模型设计思路

### 1.3 非目标（本期不做）

- 不对接真实企业系统，不使用任何公司真实数据
- 不做完整的组织架构管理后台（部门/门店层级权限先简化）
- 不做审核发布工作流（知识变更审批流程留到后续迭代）
- 不做语音入口（ASR/TTS）

## 二、用户角色与场景

### 2.1 角色定义

| 角色 | 说明 | 可见知识范围 |
|---|---|---|
| 客服（customer_service） | 一线电话/在线客服 | 仅"公开级"知识：标准话术、通用FAQ |
| 技师（technician） | 门店维修技师 | "公开级" + "内部级"：完整技术手册、诊断步骤 |
| 工程师（engineer） | 区域技术支持 | 全部级别，含"机密级"：未公开质量问题、内部处理方案 |
| 管理员（admin） | 知识库维护者 | 全部级别 + 内容管理权限（增删改知识条目） |

### 2.2 核心使用场景

**场景 A：客服快速答疑**
客服收到客户报修电话，输入故障描述，系统返回标准话术建议，不暴露内部诊断细节。

**场景 B：技师现场诊断**
技师在店内根据故障现象查询，系统返回完整诊断步骤和维修方案，可按车型/品牌过滤。

**场景 C：工程师查疑难案例**
工程师查询是否存在已知的批次性问题，系统需要能检索到机密级内容（如果该工程师有权限）。

**场景 D：越权测试（验证用）**
用客服账号尝试查询机密级内容，验证系统能正确拒绝返回或做降级处理，而不是"检索到了但没显示"这种表面处理。

### 3.2 权限模型详细设计

#### 3.2.1 密级定义

| 密级 | 说明 | 示例 |
|---|---|---|
| public | 公开级，无限制 | 通用故障码解释、客户话术 |
| internal | 内部级，需登录角色为技师及以上 | 完整维修手册、诊断步骤 |
| confidential | 机密级，仅工程师及以上 | 未公开质量问题、批次性缺陷处理方案 |

#### 3.2.2 权限判断逻辑

检索流程中权限过滤必须发生在**向量检索之前或检索的同一阶段**，作为前置条件参与召回，而不是等 LLM 生成完答案后再做内容审查。

```
用户发起查询
    ↓
获取用户角色 role
    ↓
根据 role 计算可见密级范围 allowed_levels
    （customer_service → [public]）
    （technician → [public, internal]）
    （engineer/admin → [public, internal, confidential]）
    ↓
向量检索时带 metadata 过滤条件：security_level IN allowed_levels
    ↓
仅在过滤后的候选集中做相似度排序，返回 Top-K
    ↓
LLM 基于过滤后的上下文生成回答
    ↓
记录本次检索的过滤条件和结果，写入访问日志
```

#### 3.2.3 越权访问处理

- 若过滤后候选集为空（说明用户在其权限范围内查不到相关知识），系统应明确告知"未查询到您权限范围内的相关信息"，而不是沉默返回空答案或编造答案
- 越权尝试（比如客服反复查询机密类关键词）需要记录日志，为后续风控分析留痕

## 四、数据设计

### 4.1 数据来源方案

由于不涉及公司真实数据，采用三层虚构+公开数据结合的方式构造数据集：

| 数据层 | 内容 | 规模 | 对应密级 |
|---|---|---|---|
| 公开层 | 公开 OBD-II 故障码解释、通用维修科普（自行改写，非转载） | 30-50 条 | public |
| 内部层 | 虚构品牌/车型的故障案例（诊断步骤+维修方案） | 60-100 条 | internal |
| 机密层 | 虚构的批次性质量问题/内部处理方案 | 10-15 条 | confidential |
| 结构化层 | 模拟备件库存表（非向量检索，走结构化查询） | 20-30 条记录 | 按角色可查 |

虚构车型体系示例（可自行替换命名）：
- 品牌"极狼"（越野系列）
- 品牌"云澈"（新能源系列）
- 品牌"星阈"（轿车系列）

每个品牌下设 2-3 个虚构车型，每车型 10-15 条知识案例。

### 4.2 知识条目数据结构（Schema）

```json
{
  "id": "case_001",
  "brand": "极狼",
  "model": "极狼-X7",
  "category": "刹车系统",
  "title": "刹车异响故障排查",
  "content": "故障现象：...\n诊断步骤：...\n维修方案：...",
  "security_level": "internal",
  "role_visible": ["technician", "engineer", "admin"],
  "source_type": "manual",
  "created_at": "2026-01-15",
  "status": "active"
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| security_level | 密级：public / internal / confidential |
| role_visible | 冗余字段，明确列出可见角色，便于直接过滤 |
| source_type | 知识来源类型：manual（手册）/ bulletin（技术公告）/ faq（客服话术）|
| status | active / deprecated，deprecated 内容检索时默认降权或排除 |

### 4.3 数据生成方式

建议用 LLM 辅助批量生成虚构知识条目（可用你已在用的 GLM-4-Flash / Kimi 等免费 API），人工只需定义字段结构、编写生成 prompt 模板，并抽样审核内容合理性，不需要逐条手写。

## 五、技术方案

### 5.1 技术栈

- 后端：Node.js + NestJS + TypeScript
- 向量数据库：Milvus（Zilliz Cloud 免费层 或 Milvus Lite 本地）
- 结构化数据：PostgreSQL / SQLite（模拟备件库存、用户角色表）
- LLM API：GLM-4-Flash / Kimi / 其他免费国内 API（用于生成回答和数据构造）
- 前端：Vue 3 + TS（复用你已熟悉的技术栈）
- Agent 编排：LangGraph（多步骤路由：判断查询类型 → 决定走向量检索还是结构化查询 → 生成回答）
- 模型调用与流式通信：**Vercel AI SDK**（负责与 LLM 的实际通信、流式输出、前后端消息协议，见 5.2.1）

### 5.2 系统架构（简化版）

```
┌─────────────────────┐
│      Vue3 前端        │
│  @ai-sdk/vue useChat  │  角色切换（模拟登录）+ 问答界面 + 溯源展示
│  流式渲染/loading状态  │
└──────────┬───────────┘
           │ SSE 流式响应
┌──────────▼───────────────────┐
│       NestJS 后端 API          │
│  ┌──────────────────────────┐ │
│  │   权限中间件（角色解析）    │ │
│  └──────────────────────────┘ │
│  ┌──────────────────────────┐ │
│  │   LangGraph Agent 编排层   │ │
│  │  （查询意图判断/路由/状态）  │ │
│  │        │                  │ │
│  │        ▼                  │ │
│  │  Vercel AI SDK             │ │
│  │  streamText/generateObject │ │  ← 实际调用 LLM、结构化输出、
│  │  （每个节点内部调用模型）    │ │     流式转发给前端
│  └────┬──────────────┬───────┘ │
└───────┼──────────────┼─────────┘
        │              │
┌───────▼──────┐ ┌─────▼────────┐
│ Milvus 向量库 │ │ PostgreSQL   │
│ (知识+权限    │ │ (备件库存/   │
│  metadata)   │ │  用户角色表)  │
└──────────────┘ └──────────────┘
```

### 5.2.1 Vercel AI SDK 在本项目中的定位与能力

**分工原则**：LangGraph 负责"流程编排"（多步骤路由、状态管理、决定走哪条分支），Vercel AI SDK 负责"模型通信"（每个节点内部真正跟 LLM 对话的部分、流式输出、前后端协议）。两者不是二选一，是配合使用。

Vercel AI SDK 是框架无关的 TypeScript 工具包，核心包 `ai` 不依赖 Next.js，可以直接在 NestJS 中当普通 npm 包使用；前端有 `@ai-sdk/vue` 提供 Vue 3 组合式 API。对本项目有直接价值的能力：

| 能力 | 对应本项目的用途 |
|---|---|
| `streamText` | 流式生成回答，边生成边推给前端，提升问答体验（不用等完整回答生成完才显示） |
| `generateObject` + Zod schema | 让 LLM 输出类型安全的结构化 JSON，用于 LangGraph 路由节点判断意图，比如约束模型只能返回 `{ intent: "knowledge_search" \| "inventory_query" }`，避免自己写正则/关键词解析意图 |
| 统一 Provider 接口 | 切换 GLM-4-Flash / Kimi / 其他模型基本只改配置，不用重写调用逻辑，方便后续做多模型效果对比 |
| 工具调用（tool calling） | 封装了函数注册、参数校验、调用链管理，可用于对接结构化数据查询（备件库存）这类"工具" |
| `@ai-sdk/vue` 的 `useChat` | 前端自动管理消息状态、流式渲染、loading/错误态，省去手写 SSE 解析和打字机效果的工作量 |

**在本项目里具体怎么落地**：
- LangGraph 的"查询意图路由"节点：用 `generateObject` 约束输出格式，比纯 prompt 判断更可靠
- LangGraph 的"回答生成"节点：用 `streamText` 调用模型并流式返回，NestJS 侧透传 SSE 给前端
- 前端问答界面：用 `useChat` 接管消息列表、输入框状态、流式展示，不用自己维护这套状态机
- 权限过滤逻辑仍然在 LangGraph 编排层和检索阶段完成，AI SDK 不参与权限判断，只负责"过滤后的内容怎么生成回答、怎么传给前端"

### 5.3 关键实现要点

- 向量检索调用时，metadata filter 需要支持按 `security_level` 和 `role_visible` 过滤（Milvus 支持标量字段过滤，检索时带条件）
- LangGraph 中设计一个路由节点，判断用户问题是"知识检索类"还是"库存/工单查询类"，分别走不同工具（意图判断本身用 AI SDK 的 `generateObject` 实现）
- 每次回答需要返回引用的知识条目 id 和标题，前端展示溯源信息
- 用户角色通过模拟登录（下拉框选择角色）实现，不做真实鉴权系统，聚焦在权限过滤逻辑本身
- 流式回答与溯源信息的展示需要协调：可以先流式输出正文，末尾再补一个结构化的引用列表（用 AI SDK 的工具调用或消息附加数据机制实现）

## 六、里程碑规划

| 阶段 | 目标 | 交付物 |
|---|---|---|
| M1：数据与基础检索 | 构造数据集，跑通基础向量检索问答（暂不接权限） | 100+ 条知识数据，能检索问答的 Demo |
| M2：权限过滤 | 接入角色权限模型，实现检索阶段权限过滤 | 角色切换后检索结果不同，越权测试通过 |
| M3：Agent 编排 | 接入 LangGraph，支持结构化数据查询路由 | 能区分"查知识"和"查库存"两类问题 |
| M4：体验完善 | 溯源展示、多轮记忆、前端优化 | 完整可演示的 Demo |
| M5（可选）：进阶 | 图谱检索、反馈闭环 | 视时间精力决定是否推进 |

## 七、验收标准（MVP）

- [ ] 客服角色查询机密级内容问题，系统不返回机密信息，且明确提示权限不足
- [ ] 技师角色能查到公开+内部级知识，查不到机密级
- [ ] 工程师角色能查到全部密级知识
- [ ] 每次回答能展示引用的知识来源（条目标题/id）
- [ ] 结构化查询（如"XX车型备件库存"）能正确路由到数据库查询而非向量检索
- [ ] 越权查询行为有日志记录
